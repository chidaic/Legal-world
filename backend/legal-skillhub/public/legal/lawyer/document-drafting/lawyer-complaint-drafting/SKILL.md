# 目标

根据玩家律师提供的提示、草稿和案件上下文，辅助生成可供玩家继续编辑确认的《民事起诉状》正文。

本 Skill 只规定文书内容、结构和质量要求。玩家最终确认、保存结果文件和导出 PDF 由后端流程处理，不要在正文中写工具调用、PDF 路径、导出说明或完成提示。

# 生成规则

1. 只输出完整《民事起诉状》正文，不输出解释、摘要、Markdown 标题、代码块或操作说明。
2. 以玩家提示和玩家草稿为主要意图来源；案件上下文只用于补足事实、主体、诉求、证据和法院信息。
3. 不要编造事实、证据、金额、日期、地址、身份证号或主体信息。
4. 信息缺失时，在对应位置写“待补充”，不要为了完整而虚构。
5. 诉讼请求必须具体、可执行，并与事实和理由对应。
6. 事实和理由应围绕法律关系、履行情况、违约或侵权事实、损失或请求依据展开，不要写成聊天总结。
7. 证据部分只列已经从玩家提示、草稿或案件上下文中出现的证据；证据名称、来源或证明目的不明时写“待补充”。
8. 如果玩家草稿已有明确表达，优先保留其核心诉求和事实逻辑，只做结构化、法律化和格式化整理。

# 文书模板

正文必须按下列模板组织：

```text
                    民事起诉状

原告（自然人）：XXX，男/女，XXXX年XX月XX日生，X族，住XXXXXX。
原告（法人）：XXX，法定代表人：XXX，住所：XXXXXX。

被告（自然人）：XXX，男/女，XXXX年XX月XX日生，X族，住XXXXXX。
被告（法人）：XXX，法定代表人：XXX，住所：XXXXXX。

诉讼请求：
1. 诉讼请求1。
2. 诉讼请求2。
...

事实和理由：
...

证据和证据来源，证人证言等：
1. 证据名称、来源及证明目的。
...

此致
[受诉法院名称]
```

# 质量检查

输出前检查：

- 标题是否为“民事起诉状”。
- 原告、被告、诉讼请求、事实和理由、证据、受诉法院是否齐全。
- 诉讼请求与事实理由是否对应。
- 是否误写了工具调用、PDF、文件路径、后台导出或“起草结束”等流程性内容。
- 是否存在没有来源的事实、金额、日期、证据或主体信息。

# 兼容说明

如果本 Skill 被自动 Agent 场景加载，是否调用文书工具由场景系统规则决定；本 Skill 不重复规定工具调用流程。玩家律师辅助接口使用本 Skill 时，只把它作为正文生成规范。
